BluesMinds automatically selects the optimal upstream provider based on **health**, **latency**, and **cost** — then normalizes the response to OpenAI format.

## How It Works

```
Your Request ──▶ BluesMinds Gateway
                      │
                      ├──▶ Channel A (OpenAI Direct)      ✅ Primary
                      ├──▶ Channel B (Azure OpenAI)        ⚡ Fallback
                      └──▶ Channel C (Third-party proxy)   💰 Cost-optimized
```

When you call `POST /v1/chat/completions`, BluesMinds:

::steps
### Finds all channels serving your requested model

### Picks the optimal channel based on health, latency, and cost

### Normalizes the response to OpenAI format

### Retries on alternate channels if one fails
::

## Failover Triggers

The gateway automatically fails over on:

- **5xx** upstream errors
- **Network timeouts**
- **Unresolvable `429`** rate limits

Failover is **transparent** — your app always receives the same response shape regardless of which channel served it.

## Load Balancing

When multiple channels are healthy, BluesMinds balances across them for:

- **Lower latency** — favors channels with low p50 response times
- **Rate-limit distribution** — spreads requests to avoid per-provider limits
- **Cost efficiency** — selects cheaper channels when quality is equivalent

## Model Availability

| Status | Meaning |
|--------|---------|
| Model in `/v1/models` | At least one healthy channel exists |
| Model absent from list | No channel currently available |
| `404` on inference | Channel dropped mid-request (rare) |

## Channel Listing

View available channels via the Management API:

```bash
curl https://api.bluesminds.com/api/channel/ \
  -H "Authorization: <session_token>"
```

## Best Practices

::callout{type="info" icon="lucide:lightbulb"}
- Regularly check `/v1/models` for availability changes
- Don't retry `404` errors — no channel serves that model
- Use exponential backoff on `429` when all channels are exhausted
::
